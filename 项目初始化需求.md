### 项目整体定位

我们是一家石油化工行业的设计院，正在设计**一个用于“设计流程再造 / 交付标准化”的轻量级数据字典系统**

前后端分离，核心技术栈用 ReactFlow、Next.js、pnpm、Shadcn、Tailwind、Zustand、jotai、Supabase

因为流程可视化是这个平台的核心交互，可以参考 Dify、Langflow 等这些使用reactflow的项目，补充可能用到的技术栈

### 核心业务功能(暂时的思路，可以帮我优化补充)


- **数据字典与数据治理**
  - 维护统一的数据字典：字段编码、名称、类型、长度、是否可空、是否唯一等。
  - 管理**数据源**（字段从哪里来）和**数据流向**（数据流向哪里），支持多条件查询。
  - 维护字段间的**依赖关系**（计算、拼接、转换、派生、查找、条件等），并自动检测循环依赖。
  - 支持字段生命周期管理：草稿 / 已发布 / 已作废，用于规范全平台的数据标准。

- **交付模型 / 流程建模**
  - 提供一个可视化“交付流程图”/“设计流程图”的建模能力：节点 + 连线。
  - 把流程模型（节点、连接、视图缩放/偏移等）**持久化到数据库**，可获取最新版本。
  - 用于表达从“数据源 → 流程 → 输出/交付物”的标准化路径，为设计流程再造提供依据。

- **项目与项目字段管理**
  - 管理项目实体（项目 CRUD），以及**项目与数据字典字段**的关联（特别是按数据流向）。
  - 用统一的数据字典字段来描述各项目的输入、处理、输出，为后续报表、集成、自动化打基础。

- **设计交付文档管理**
  - 围绕“交付模型 / 数据流向”，管理项目下的“设计交付文档”（草稿 / 已发布版本、操作记录等）。


总结起来，打造成一个**“围绕设计流程与数据流向的数据标准平台”**：  
既管理各种文档和交付文件本身，又管理这些文档背后的字段标准、数据源/流向和交付流程模型，用来支撑我们在“设计流程再造”中的规范化和自动化。
